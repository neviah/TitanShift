## 1. Goals

The Tool Module must:

- Provide a **unified, safe interface** for all tools (local, remote, CLI, browser, etc.).
- Power **skills and agents** without hardãcoding specific tools.
- Support:
  - **Core system tools** (filesystem, shell, runners, logging, config, graph, memory).
  - **Productivity tools** (code, text, PDF, archive, git, ffmpeg, OCR, etc.).

- Be compatible with:
  - **CLIãAnything** style tools: `https://github.com/HKUDS/CLI-Anything`

## 2. Tool model and schema

### 2.1 ToolDefinition

```ts
type ToolDefinition = {
  id: string
  name: string
  description: string
  category: 'core' | 'productivity' | 'optional' | 'external'
  tags: string[]                 // ['filesystem', 'cli', 'http', 'ocr', 'browser', ...]
  inputSchema: any               // JSON schema
  outputSchema: any              // JSON schema
  implementation: ToolImplementation
  permissions: ToolPermissions
  sandbox: ToolSandboxConfig
  metadata?: ToolMetadata
}
```

### 2.2 ToolImplementation

```ts
type ToolImplementation =
  | { type: 'builtin'; handler: (input: any, context: ToolContext) => Promise<any> }
  | { type: 'cli'; command: string; argsTemplate?: string[] }
  | { type: 'http'; url: string; method: 'GET' | 'POST' | 'PUT' | 'DELETE'; headers?: Record<string, string> }
  | { type: 'script'; language: 'python' | 'node'; code: string }
```

### 2.3 ToolPermissions

```ts
type ToolPermissions = {
  requiresUserConsent: boolean
  allowedInAutonomousMode: boolean
  allowedDomains?: string[]      // ['code', 'research', 'automation', ...]
  forbiddenForRoles?: string[]   // ['guest', ...]
}
```

### 2.4 ToolSandboxConfig

```ts
type ToolSandboxConfig = {
  filesystemAccess: 'none' | 'read' | 'readwrite'
  networkAccess: boolean
  maxRuntimeMs: number
  maxOutputSize: number
}
```

### 2.5 ToolMetadata

```ts
type ToolMetadata = {
  version?: string
  author?: string
  repoUrl?: string
  installCommand?: string
  requiresBinary?: boolean
}
```

### 2.6 ToolContext

```ts
type ToolContext = {
  agentId?: string
  taskId?: string
  userId?: string
  memoryApi: MemoryAPI
  logger: Logger
  config: ConfigManager
}
```

## 3. Tool Registry

### 3.1 Responsibilities

- Register tools.
- Look up tools by ID/name/tags.
- Enforce permissions and sandboxing.
- Provide discovery for skills and orchestrator.
- Support local + marketplace tools.

### 3.2 API

```ts
type ToolRegistryAPI = {
  registerTool(tool: ToolDefinition): void
  unregisterTool(toolId: string): void
  getTool(toolId: string): ToolDefinition | null
  listTools(filter?: ToolFilter): ToolDefinition[]
  executeTool(toolId: string, input: any, context: ToolContext): Promise<any>
}
```

```ts
type ToolFilter = {
  category?: 'core' | 'productivity' | 'optional' | 'external'
  tags?: string[]
  allowedForDomain?: string
}
```

### 3.3 Execution flow

1. Orchestrator/agent/skill calls `executeTool(toolId, input, context)`.
2. Registry:
   - Fetches `ToolDefinition`.
   - Checks permissions.
   - Applies sandbox config.
   - Dispatches to implementation.
3. Logs:
   - toolId, input summary, output summary, errors.
4. Emergency Module can hook into failures.

## 6. Tool selection logic (orchestrator + skills)

### 6.1 For a given task

1. Orchestrator analyzes task ã `TaskProfile`.
2. Skill selector picks skills.
3. Tool selector:
   - Collects tools required by skills.
   - Adds supporting tools (logging, filesystem, etc.).
   - Filters by:
     - availability  
     - permissions  
     - sandbox rules  

### 6.2 APIãanything & CLIãanything skills

We define **skills** that wrap generic tools:

- `ApiAnythingSkill` ã uses `HttpFetchTool`.
- `CliAnythingSkill` ã uses `ShellTool` / `CLIAnythingTool` / `OpenCLITool`.
- `WebCrawlSkill` ã uses `crawl4aiTool` / PlaywrightTool.

Agents donãt need to know the internalsãthey just call the skill.

## 7. Permissions, safety, and sandboxing

- Tools are **not equal**ãeach has:
  - permissions  
  - sandbox config  
  - risk level  

- Autonomous mode:
  - Some tools are disabled unless explicitly allowed (e.g., ShellTool, GitTool, PlaywrightTool).

- Emergency Module:
  - Can disable tools that repeatedly fail or cause issues.
  - Can suggest disabling risky tools.

## 8. Integration with other modules

### 8.1 Skills

- Skills declare required tools by ID.
- Tool Registry ensures they exist.
- If missing:
  - Skill is degraded or disabled.
  - Emergency Module can suggest installing tools.

### 8.2 Orchestrator

- Uses Tool Registry to:
  - select tools for agents.
  - enforce tool budgets (maxToolCalls).
  - log tool usage.

### 8.3 Memory/Graph

- Tool usage is recorded:
  - Episodic memory: ãTool X used in Task Y.ã
  - Graph memory:
    - `skill --uses--> tool`
    - `tool --failed_with--> pattern`
- This enables:
  - better tool selection over time.
  - avoiding tools that fail in similar contexts.

### 8.4 Emergency Module

- On tool failure:
  - Receives `TOOL_ERROR`.
  - Runs diagnostics.
  - May propose:
    - disabling tool  
    - updating config  
    - switching to alternative tool
