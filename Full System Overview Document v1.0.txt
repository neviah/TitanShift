# **1. Philosophy**

The Universal Harness is designed around a set of principles that ensure it remains:

- modular  
- resilient  
- extensible  
- Python‑native  
- graph‑aware  
- skill‑centric  
- user‑friendly  
- self‑diagnosing  

These principles guide every module, interface, and design decision.

---

## **1.1 Python‑First Architecture**
The entire harness is built around Python’s strengths:

- dynamic module loading (`importlib`)  
- async orchestration (`asyncio`)  
- plugin systems  
- rich ecosystem (FastAPI, APScheduler, networkx, sqlite3)  
- easy tool wrapping (CLI, HTTP, Python scripts)  
- perfect for LLM workflows  

Python is the backbone of the runtime, orchestrator, skills, tools, memory, and emergency systems.

---

## **1.2 Modularity Above All**
Every component — tools, skills, agents, memory layers, UI panels — is a module.

Modules can be:

- added  
- removed  
- replaced  
- upgraded  

…without breaking the system.

---

## **1.3 Failure Isolation**
No single module can take down the system.

If a tool crashes → isolate it.  
If a skill misbehaves → disable it.  
If an agent loops → terminate it.  
If a module fails → Emergency Module intervenes.

The harness must **never** crash as a whole.

---

## **1.4 Skill‑Centric Intelligence**
The harness does not rely on “magic prompts.”

It relies on **skills**:

- reusable  
- composable  
- testable  
- discoverable  
- graph‑indexed  

Agents become powerful because skills are powerful.

---

## **1.5 Graph‑Aware Reasoning**
The system understands itself.

Everything becomes a graph node:

- skills  
- tools  
- modules  
- agents  
- tasks  
- errors  
- fixes  
- preferences  

Edges represent:

- uses  
- depends_on  
- caused_by  
- fixed_by  
- belongs_to  

This enables:

- smarter tool selection  
- better skill matching  
- pattern recognition  
- self‑improvement  

---

## **1.6 Local‑First, Cloud‑additional**
The harness works with local models via ollama and lmstudio.

Cloud models and APIs are additional enhancements, not dependencies.
must work with claude code, openai compatible, openrouter, etc.

Local‑first means:

- local LLMs (LM Studio, Ollama, GGUF)  
- local vector DB  
- local graph backend  
- local tools  
- local skills  

---

## **1.7 Reproducibility**
Every task, agent, skill, tool call, and memory update is logged and traceable.

Users can:

- replay workflows  
- audit decisions  
- inspect memory  
- view agent reasoning  

---

## **1.8 UI‑Driven Accessibility**
Non‑technical users can configure:

- models  
- tools  
- skills  
- autonomy  
- emergency behavior  
- memory scopes  

…all from a clean, modern web UI.

---

## **1.9 Self‑Diagnosis and Optional Self‑Healing**
The Emergency Module acts like a “fire department”:

- detects failures  
- spawns diagnostic agents  
- proposes fixes  
- optionally auto‑repairs  

---

## **1.10 Human‑Centered Design**
The system is built for:

- creators  
- developers  
- researchers  
- automation builders  
- non‑technical users  

It must feel intuitive, powerful, and safe.

---

# **4.1 Runtime Layer**

The Runtime Layer is the foundation of the harness.  
It provides:

- dynamic module loading  
- configuration  
- event routing  
- lifecycle management  

Everything else sits on top of it.

---

## **4.1.1 Module Loader (Python Plugin System)**

### **Responsibilities**
- Discover Python modules in `modules/` directory.
- Load them dynamically using `importlib`.
- Hot‑reload modules using `watchdog`.
- Register modules with the system.

### **Python‑Native Behavior**
- Each module is a Python package with:
  - `__init__.py`
  - `module.py` exporting:
    ```python
    def register(runtime): ...
    ```

### **Interfaces**
```python
class ModuleLoader:
    def register_module(self, name: str, capabilities: list[str], hooks: dict): ...
    def list_modules(self) -> list[ModuleInfo]: ...
    def reload_module(self, name: str): ...
```

### **Failure Isolation**
- If a module fails to load:
  - Log error
  - Notify Emergency Module
  - Continue running without that module

---

## **4.1.2 Config Manager**

### **Responsibilities**
- Central configuration store.
- Merge config from:
  - YAML/JSON files
  - environment variables
  - UI overrides

### **Python‑Native Behavior**
- Uses `pydantic` or `dataclasses` for typed config.
- Supports hierarchical scopes.

### **Interfaces**
```python
class ConfigManager:
    def get(self, key: str): ...
    def set(self, key: str, value): ...
    def get_scoped(self, scope: str, key: str): ...
```

---

## **4.1.3 Event Bus**

### **Responsibilities**
- Decoupled communication between modules.
- Async pub/sub.

### **Python‑Native Behavior**
- Uses `asyncio.Queue` or custom event loop.
- Handlers are async functions.

### **Interfaces**
```python
class EventBus:
    async def publish(self, event_type: str, payload: dict): ...
    def subscribe(self, event_type: str, handler: Callable): ...
```

# **4.2 Model Module (LLM Adapter)**

### **Responsibilities**

- Support multiple backends:
  - OpenAI
  - Claude
  - DeepSeek
  - Local GGUF (via llama.cpp bindings)
  - LM Studio
  - Ollama

### **Python‑Native Behavior**

### **Interfaces**

```python
class ModelAdapter:

    def estimate_tokens(self, text: str) -> int: ...
```

### **Failure Isolation**

  - Try fallback chain

# **4.3 Tool Module (Tool Registry)**

### **Responsibilities**
- Register tools.
- Execute tools safely.
- Enforce sandboxing.
- Provide discovery for skills and agents.

### **Python‑Native Behavior**
Tools are Python classes:

```python
class Tool:

    output_schema: dict
    permissions: dict

        ...
```

Tools can be:

- builtin Python functions  
- wrappers around CLI commands  
- wrappers around HTTP APIs  
- wrappers around Python scripts  

### **Interfaces**
```python
class ToolRegistry:
    def register_tool(self, tool: Tool): ...
```

### **Failure Isolation**
- Tool errors do not crash agents.

# **4.5 Memory Module (Multi‑Layer + Graph)**

### **Responsibilities**
Provide six memory layers:

1. Working memory  
2. Short‑term memory  
3. Long‑term memory  
4. Semantic memory  
5. Graph memory  
6. Episodic memory  

### **Python‑Native Behavior**
- Working memory → dict  
- Short‑term → SQLite or JSON  
- Long‑term → Markdown/JSON  
- Semantic → embeddings + SQLite  
- Graph → networkx or custom Python graph backend  
- Episodic → SQLite  

### **Interfaces**
```python
class Memory:
    def save_fact(self, scope: str, fact: dict): ...
    def query_facts(self, query: dict) -> list[dict]: ...
    async def embed_and_store(self, text: str, metadata: dict) -> str: ...
    async def semantic_search(self, query: str, filters: dict) -> list[dict]: ...
```

Graph memory:

```python
class GraphMemory:
    def add_node(self, node: GraphNode): ...
    def add_edge(self, edge: GraphEdge): ...
    def query(self, pattern: GraphQuery) -> list[GraphResult]: ...
```

### **Failure Isolation**
- Memory corruption triggers Emergency diagnostics.
- Graph backend can fall back to lightweight mode.

# **4.6 State Machine Module**

### **Responsibilities**
- Define agent loop:
  - PLAN → ACT → REFLECT
- Enforce budgets:
  - steps
  - tokens
  - duration
  - tool calls
  - subtasks
- Handle errors.

### **Python‑Native Behavior**
Async agent loop:

```python
async def run_agent(self, agent):
    while not agent.done:
        action = await self.plan(agent)
        result = await self.act(agent, action)
        await self.reflect(agent, result)
```

### **Interfaces**
```python
class StateMachine:
    async def run_task(self, task: TaskDefinition, context) -> TaskResult: ...
    async def next_step(self, state: AgentState) -> AgentAction: ...
```

### **Failure Isolation**
- If agent exceeds budget:
  - terminate agent
  - notify orchestrator

# **4.8 Execution Module (Python Execution Layer)**

The Execution Module is responsible for safely running:

- Python code  
- Node.js code  
- Shell commands  
- Browser automation (optional)  
- Any external process  

It is the “hands” of the harness — the part that actually performs actions in the real world.

---

## **4.8.1 Responsibilities**

- Provide a unified interface for executing code and commands.
- Enforce sandboxing and resource limits.
- Capture stdout, stderr, exit codes, and exceptions.
- Support async execution.
- Integrate with the Tool Module (tools call into this module).
- Provide safe wrappers around:
  - Python execution  
  - Node execution  
  - Shell commands  
  - Browser automation (Playwright/Lightpanda)  

---

## **4.8.2 Python‑Native Behavior**

### **Python Execution**
Uses:
- `asyncio.create_subprocess_exec`
- or direct `exec()` inside a sandboxed namespace

### **Shell Execution**
Uses:
- `asyncio.create_subprocess_shell`

### **Node Execution**
Uses:
- `node` binary via subprocess

### **Browser Automation**
Optional tools wrap:
- Playwright Python bindings  
- Lightpanda browser  

---

## **4.8.3 Interfaces**

```python
class ExecutionModule:
    async def run_code(self, language: str, code: str, context) -> ExecutionResult: ...
    async def run_command(self, command: str, args: list[str], context) -> ExecutionResult: ...
    async def run_python(self, code: str, context) -> ExecutionResult: ...
    async def run_node(self, code: str, context) -> ExecutionResult: ...
```

---

## **4.8.4 Failure Isolation**

- Timeouts enforced via `asyncio.wait_for`
- Memory limits enforced via subprocess flags
- Crashes do not propagate to agents
- Errors routed to Emergency Module

---

# **4.9 Emergency Module (Diagnostics + Self‑Healing)**

The Emergency Module is the harness’s **fire department**.

It detects failures, diagnoses them, and proposes fixes.

---

## **4.9.1 Responsibilities**

- Detect failures from:
  - Event Bus  
  - Scheduler heartbeats  
  - Tool errors  
  - Agent crashes  
  - Module exceptions  
- Spawn diagnostic agents
- Aggregate diagnoses
- Propose fixes
- Optionally auto‑repair (user‑approved)

---

## **4.9.2 Python‑Native Behavior**

- Diagnostic agents are spawned via the Orchestrator.
- Each diagnostic agent receives:
  - logs  
  - memory context  
  - graph nodes  
  - failure event  
- Agents produce:
  - hypothesis  
  - probable cause  
  - confidence  
  - suggested fixes  

---

## **4.9.3 Interfaces**

```python
class EmergencyModule:
    async def on_failure(self, event: FailureEvent): ...
    async def run_diagnostics(self, event: FailureEvent) -> list[Diagnosis]: ...
    def propose_fixes(self, diagnoses: list[Diagnosis]) -> FixPlan: ...
    async def apply_fix(self, fix_plan: FixPlan): ...
```

---

## **4.9.4 Failure Isolation**

- Emergency Module is the last line of defense.
- If Emergency Module fails:
  - Log critical error
  - System enters “safe mode”
  - Only minimal chat agent remains active

# **5. Cross‑Module Flows**

This section explains how the modules interact during real operations.  
These flows are essential for Copilot to scaffold correct integrations.

---

# **5.1 Flow: User Sends a Chat Message**

```
User → UI → API → Orchestrator → Agent → State Machine
                                   ↓
                              Skill Module
                                   ↓
                              Tool Module
                                   ↓
                              Execution Module
                                   ↓
                                Memory
                                   ↓
                             UI (response)
```

### **Step‑by‑step**

1. **User types a message** in the UI.  
2. UI sends it to **FastAPI** (`POST /chat`).  
3. API forwards it to the **Orchestrator**.  
4. Orchestrator:
   - selects or spawns an agent  
   - loads skills/tools  
   - builds an `AgentConfig`  
5. Agent enters the **State Machine** loop:
   - PLAN → ACT → REFLECT  
6. Skills may be invoked.  
7. Skills may call tools.  
8. Tools may call the Execution Module.  
9. Memory is updated:
   - working  
   - short‑term  
   - episodic  
   - graph  
10. Final response is returned to the UI.  
11. UI displays the message and updates the visualization.

---

# **5.2 Flow: Tool Execution**

```
Skill → Tool Module → Execution Module → Tool Module → Skill → Agent
```

### **Step‑by‑step**

1. Skill calls `execute_tool()`.  
2. Tool Module:
   - validates input schema  
   - checks permissions  
   - applies sandbox rules  
3. Execution Module runs:
   - Python code  
   - Node code  
   - shell commands  
   - browser automation  
4. Output is returned to the Skill.  
5. Skill returns result to Agent.  
6. Agent continues loop.

---

# **5.3 Flow: Emergency Detection**

```
Event Bus → Emergency Module → Diagnostic Agents → Fix Plan → User Approval → Apply Fix
```

### **Step‑by‑step**

1. A failure event is published:
   - tool crash  
   - agent crash  
   - heartbeat failure  
   - module exception  
2. Emergency Module receives event.  
3. It spawns diagnostic agents.  
4. Each agent:
   - reads logs  
   - queries memory  
   - inspects graph  
   - produces diagnosis  
5. Emergency Module aggregates results.  
6. Proposes a fix plan.  
7. User approves fix.  
8. Fix is applied.

---

# **5.4 Flow: Scheduled Job**

```
Scheduler → Orchestrator → Agent → Tools/Skills → Memory → Logs
```

# **Appendix A — Orchestrator Deep Dive**

The Orchestrator is the “brain” of the harness.  
It decides:

- which agent to spawn  
- which skills to use  
- which tools to allow  
- how to allocate memory  
- how to enforce budgets  
- how to coordinate multi‑agent workflows  

This appendix defines its internal architecture.

---

## **A.1 Responsibilities**

1. Interpret tasks  
2. Select or spawn agents  
3. Assign skills and tools  
4. Manage agent lifecycles  
5. Coordinate subtasks  
6. Enforce budgets  
7. Communicate with the State Machine  
8. Integrate with Memory, Tools, Skills, and Emergency modules  

---

## **A.2 Python Architecture**

```
orchestrator/
    __init__.py
    orchestrator.py
    agent_factory.py
    task_manager.py
    planner.py
```

---

## **A.3 Agent Creation Flow**

```
Task → Orchestrator → AgentFactory → AgentConfig → Agent
```

### **Step‑by‑step**

1. Orchestrator receives a task.  
2. It queries the Skill Registry for relevant skills.  
3. It queries the Tool Registry for required tools.  
4. It builds an `AgentConfig`.  
5. It calls `AgentFactory.create_agent(config)`.  
6. Agent is returned and passed to the State Machine.

---

## **A.4 AgentConfig Construction**

```python
config = AgentConfig(
    role="Researcher",
    skills=["web_search", "summarize"],
    tools=["http_fetch", "browser_automation"],
    model_profile={"model": "deepseek-chat", "temperature": 0.2},
    memory_scope={"working": True, "short_term": True, "long_term": False},
    budget={"steps": 20, "tokens": 8000, "tool_calls": 10}
)
```

---

## **A.5 Multi‑Agent Coordination**

Agents can spawn subtasks:

```
Parent Agent → Orchestrator → Child Agent
```

The Orchestrator:

- creates child agents  
- tracks dependencies  
- merges results  
- updates graph memory

## **A.6 Failure Handling**

If an agent fails:

- Orchestrator logs the failure  
- Publishes `AGENT_CRASHED` event  
- Emergency Module takes over

# **Appendix B — Agent Lifecycle Deep Dive**

This appendix defines the internal loop of every agent.

---

## **B.1 Lifecycle Overview**

```
SPAWN → INITIALIZE → PLAN → ACT → REFLECT → LOOP/END
```

---

## **B.2 Python Implementation Sketch**

```python
class AgentRunner:
    async def run(self, agent):
        await self.initialize(agent)
        while not agent.done:
            action = await self.plan(agent)
            result = await self.act(agent, action)
            await self.reflect(agent, result)
```

---

## **B.3 PLAN Phase**

Agent decides what to do next:

- call a skill  
- call a tool  
- ask the model  
- spawn a subtask  
- query memory  
- return final answer  

Uses:

- model adapter  
- skill registry  
- memory  

---

## **B.4 ACT Phase**

Executes the chosen action:

- skill execution  
- tool execution  
- model generation  
- subtask creation  

---

## **B.5 REFLECT Phase**

Agent:

- evaluates results  
- updates memory  
- logs reasoning  
- decides next step  

Reflection is stored in:

- episodic memory  
- graph memory  
- short‑term memory  

---

## **B.6 Termination Conditions**

Agent stops when:

- result is produced  
- budget exceeded  
- error escalated  
- emergency intervention  
- user cancels task
